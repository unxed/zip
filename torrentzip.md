# TorrentZip Format Specification (Version 1.0)

## 1. Abstract
This document defines the technical specification for **TorrentZip (trrntzip)**, a deterministic profile of the ZIP file format.
Its primary goal is to ensure that if any two archiving tools add the exact same files and directories to their archives, they produce the bit-exact same ZIP files, independent of the operating system, platform, or creation time.

## 2. Technical Requirements

### 2.1. File and Directory Ordering
- All file and directory entries must be sorted alphabetically before writing the Central Directory.
- Slashes must be normalized: all backward slashes (`\`) must be converted to forward slashes (`/`) before sorting.
- Directory paths must end with a trailing slash (`/`) before sorting.
- The sorting algorithm must perform a case-insensitive, byte-by-byte comparison of the lower-case representation of the normalized names (`strings.ToLower(name)`).

### 2.2. Timestamp Normalization
To prevent time-based discrepancies, all modification timestamps for files and directories in both the Local File Headers and Central Directory Headers must be overwritten with the fixed value:
- **Date/Time:** `11:32:00 PM on December 24, 1996` (the release date of the first MAME version).
- **MS-DOS Representation:** `ModifiedTime = 48128` (`0xbc00`), `ModifiedDate = 8600` (`0x2198`).

### 2.3. Metadata and Extra Fields Stripping
- All extra fields (`Extra` buffer) in both the Local File Headers and Central Directory Headers must be completely stripped (set to 0 bytes).
- Individual file and directory comments in the headers must be stripped (set to 0 bytes).
- Standard file attributes (External/Internal Attributes) must be reset to `0`.
- Creator Version must be set to `0` (FAT/MS-DOS compatibility), and Reader Version to `20` (Deflate compatibility).

### 2.4. Data Descriptors and Flags
- The Data Descriptor flag (bit 3 of the general purpose bit flag) is strictly prohibited. This means the `CompressedSize`, `UncompressedSize`, and `CRC32` fields must be calculated in advance and written directly into the Local File Header.
- For files, the general purpose bit flag (`Flags`) must be set to `2` (representing maximum compression).
- For directories, the general purpose bit flag (`Flags`) must be set to `0`.

### 2.5. Directory Entry Filtering
- Only **empty directories** (directories that do not act as a parent path to any other file or directory in the archive) should have explicit directory entries in the ZIP archive.
- Directories that are implied by the paths of other files must be completely omitted. For example, if the archive contains `set1/file.txt`, then the entry `set1/` is implied and must be removed. If the archive contains an empty directory `set2/` with no files inside, the entry `set2/` must be preserved.
- Directory entries must be stored using `Method = Store` (0), with `CompressedSize = 0`, `UncompressedSize = 0`, `CRC32 = 0`, and `Flags = 0`.

### 2.6. Compression Bitstream Alignment
- Non-empty files must be compressed using the `Deflate` algorithm (Method 8).
- The compressor must use maximum compression (level 9) of a standard `zlib`-compliant deflate engine. The resulting raw deflate stream must be bit-for-bit identical to the bitstream generated by the reference `zlib` library.
- Even if a file's compressed size is larger than its uncompressed size, the file must still be stored using the `Deflate` method to ensure format consistency.

### 2.7. End of Central Directory (EOCD) Comment Hash
- The archive-level comment in the End of Central Directory record is used to verify the integrity and consistency of the entire archive structure.
- The comment must consist of exactly 22 bytes in the format `TORRENTZIPPED-XXXXXXXX`.
- `XXXXXXXX` is the CRC32 checksum of the raw bytes of the Central Directory records (the segment between the End of Central Directory and the Start of Central Directory), formatted as an uppercase 8-character hexadecimal string.
